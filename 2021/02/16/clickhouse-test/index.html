ClickHouse 测试记录

发表于 2021-02-16 |

以下是使用 clickhouse 过程中做的一些对比测试，提出假设、设计实验、验证结果，做完拿到数据加深了印象，纠正一些错误的观念，才能在实际业务场景中灵活运用。

批量写入同一个分区和不同分区

  • 假设：一个批次的数据包含多个分区，写入时会产生多个 parts，影响写入效率
  • 结论：不同分区较相同分区，性能相差约 40%

LowCardinality

  • 场景：数仓开发人员在建表时，在 Hive 下习惯性的使用字符串保存各类枚举值，造成浪费。
  • 假设：在枚举值不多的情况下（小于10万），官方推荐使用类型编码，压缩效果明显，这里想看下具体压缩率如何？
  • 结论：5500万条记录，compressed 后，压缩率约为25%，效果明显

DateTime vs String

  • 场景：数仓开发人员在建表时，在 Hive 下习惯性的使用字符串保存时间
  • 假设：表中保存的时间使用原生类型与字符串有何差别，猜测是 DateTime 效率更高
  • 结论：1000万条记录，compressed 后，压缩率约为53%

排序键

  • 场景：在数据表的实际查询中，经常会有一些必选查询条件，可以将这类条件作为排序键，命中索引，加速查询
  • 准备两张表，写入 1000w 数据
  • 查询性能：没有根据 id 排序的，会全表扫描，而根据 id 排序的仅扫描 1/10 左右的数据，性能提升约 3倍，符合预期
  • 调试查询执行过程，观察读取的 parts 和索引的差异
